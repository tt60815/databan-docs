### 2.1 order_manager.py — 自動下單管理器

**檔案：** `ib_dashboard/order_manager.py`（327 行）

**功能：** 策略引擎偵測到信號後，透過 thread-safe Queue 將下單請求傳遞給 IB 線程執行。

#### 資料結構

```python
@dataclass
class OrderRequest:
    strategy_id: int        # 策略 ID
    symbol: str             # 'TSLA'
    action: str             # 'BUY' / 'SELL'
    quantity: int           # 股數
    direction: str          # 'long' / 'short'
    order_type: str         # 'entry' / 'exit'
    exit_reason: str        # 'stop_loss' / 'take_profit' / 'exit_condition' / ...
    entry_price: float      # 進場時的參考價格

@dataclass
class ManagedPosition:
    strategy_id: int
    symbol: str
    direction: str
    quantity: int
    entry_price: float      # IB fill 成交價
    entry_time: datetime
    entry_order_id: int     # IB orderId
```

#### 跨線程通訊模型

```
Strategy Thread (Databento bars)          IB Thread (_ib_thread)
         │                                       │
         │  submit_order(OrderRequest)            │
         │         │                              │
         │    ┌────▼────┐                        │
         │    │  Lock   │  ← 檢查 _inflight      │
         │    └────┬────┘                        │
         │         │                              │
         │    ┌────▼──────────────┐              │
         │    │   queue.Queue    │──────────────▶ process_queue()
         │    │  (thread-safe)   │              │       │
         │    └──────────────────┘              │  ┌────▼──────┐
         │                                      │  │ _execute_  │
         │                                      │  │  order()   │
         │                                      │  │            │
         │                                      │  │ ib.place   │
         │                                      │  │  Order()   │
         │                                      │  └────┬──────┘
         │                                      │       │
         │                                      │  ┌────▼──────┐
         │                                      │  │ _on_fill() │
         │                                      │  │ callback   │
         │                                      │  └────┬──────┘
         │                                      │       │
         │                                      │       ▼
         │                                      │  DB: ib_trades
```

#### 安全機制

| 機制 | 實作 | 說明 |
|------|------|------|
| 全域開關 | `_config['enabled']` | API 可即時啟停交易 |
| 重複防護 | `_inflight` set + Lock | 同一策略不重複下單 |
| 已持倉防護 | `_positions` dict | 同策略不重複進場 |
| 最大持倉 | `max_total_positions=5` | 最多同時持有 5 個策略的倉位 |
| 小量下單 | `default_shares=10` | 每筆 ~$2,500（TSLA） |
| IB 斷線 | re-queue | 未連線時放回 Queue 等重連 |
| 訂單取消 | `_on_cancelled` | 清除 inflight 狀態 |

#### 核心方法

```python
class OrderManager:
    def submit_order(self, req: OrderRequest)       # 策略線程呼叫（thread-safe）
    def process_queue(self)                          # IB 線程呼叫（每 1 秒）
    def _execute_order(self, req: OrderRequest)      # ib.placeOrder(Stock, MarketOrder)
    def _on_fill(self, trade, fill)                  # IB fill 回調
    def _handle_entry_fill(self, req, trade)         # 建立 ManagedPosition → 寫 DB
    def _handle_exit_fill(self, req, trade)          # 計算 PnL → 更新 DB
    def _record_entry(self, pos: ManagedPosition)    # INSERT INTO ib_trades
    def _record_exit(self, pos, exit_price, ...)     # UPDATE ib_trades SET exit_*
    def get_all_positions(self) → dict               # API 查詢用
    def get_trades_from_db(self, limit=50) → list    # 查詢交易紀錄
    def set_ib(self, ib)                             # IB 連線注入
```

### 2.2 strategy_engine.py — 即時策略評估引擎

**檔案：** `ib_dashboard/strategy_engine.py`（628 行）

**功能：** 即時評估 DB 中 `selected_strategies` 的策略條件，每根 1m bar 到達時判斷進出場。

#### 核心概念：DB 已有所有指標狀態

系統的關鍵設計：**每根 K 棒經過 `indicators.py` 計算後，已包含所有指標狀態欄位**（`k_above_50`, `dif_above_dea`, `kd_cross`, `avg_low_sell` 等，值為 1 或 -1）。策略引擎不做複雜分析，只是簡單地**比對當下這根 bar 的欄位值是否符合 DB 中定義的條件**。

```
DB 中的策略條件（觀察條件）：
[
  {"key": "entry_k_above_50",    "value": 1},    ← 進場 TF 的 K > 50
  {"key": "entry_dif_above_dea", "value": 1},    ← 進場 TF 的 DIF > DEA
  {"key": "observe_dif_above_zero", "value": 1}, ← 觀察 TF 的 DIF > 0
]

當下這根 bar 的指標值（由 indicators.py 計算）：
{
  "k_above_50": 1,       ✓ 符合
  "dif_above_dea": 1,    ✓ 符合
  "kd_cross": 1,         ← 出現 KD 金叉（進場信號）
  "avg_low_sell": 0,
  ...
}

→ 進場信號觸發 + 所有觀察條件滿足 → 進場
```

#### 每根 bar 已有的指標欄位

| 分類 | 欄位 | 意義 |
|------|------|------|
| **交叉信號** | `kd_cross` | +1=KD 金叉 / -1=KD 死叉 / 0=無 |
| | `dif_cross` | +1=MACD 金叉 / -1=MACD 死叉 / 0=無 |
| **KD 位置** | `k_above_50`, `d_above_50` | K/D 值是否 > 50 |
| | `k_above_d` | K > D |
| **MACD 位置** | `dif_above_dea` | DIF > DEA |
| | `dif_above_zero`, `dea_above_zero` | DIF/DEA > 0 |
| **純升/純降** | `dif_no_down_since_cross` | DIF 金叉後純升 |
| | `dif_no_up_since_cross` | DIF 死叉後純降 |
| | `dea_no_down_since_cross`, `dea_no_up_since_cross` | DEA 同上 |
| **J 值位置** | `j_down_1st`, `j_down_2nd` | J 值交叉後第 1/2 次下降 |
| | `j_up_1st`, `j_up_2nd` | J 值交叉後第 1/2 次上升 |
| **均高/均低** | `avg_high_buy` | 突破上一根 HA 高點 |
| | `avg_low_sell` | 跌破上一根 HA 低點 |
| **MACD 柱** | `macd_hist_turning` | MACD 柱拐點 |

> 這些欄位值都是 1 或 -1，條件檢查就是簡單的 `== 1` 比較。

#### 進出場機制

**進場信號**由交叉事件驅動：

| 進場條件 | 觸發欄位 | 值 |
|----------|----------|-----|
| KD 金叉 | `kd_cross` | 1 |
| KD 死叉 | `kd_cross` | -1 |
| MACD 金叉 | `dif_cross` | 1 |
| MACD 死叉 | `dif_cross` | -1 |

**出場信號**由反向交叉驅動（例：KD 金叉進場 → KD 死叉出場）。

**觀察條件**是額外篩選：進場信號觸發時，必須同時滿足 DB 中定義的所有觀察條件（如 `k_above_50=1`, `dif_above_dea=1`），才會真正進場。這些條件是由條件掃描系統（flask_app）找出的最佳觀察條件組合。

#### 狀態機

```
┌────────┐   進場交叉信號 +   ┌──────────┐   觀察條件全部   ┌──────────┐
│  IDLE  │   觀察條件滿足     │ WATCHING │   確認滿足       │ HOLDING  │
│ 等待中 │ ────────────────→  │  觀察中  │ ──────────────→  │  持倉中  │
└────────┘                    └──────────┘                   └──────────┘
    ▲                              │                              │
    │       Close 條件觸發         │                              │
    └──────────────────────────────┘                              │
    │                                                             │
    │             出場交叉信號觸發                                  │
    └─────────────────────────────────────────────────────────────┘
```

#### 條件評估邏輯

```python
def check_condition(row_data, cond):
    """
    對當下 bar 的指標值做簡單比較
    row_data = {"k_above_50": 1, "dif_above_dea": -1, "kd_cross": 1, ...}
    cond = {"field": "k_above_50", "operator": ">=", "value": 1}
    → row_data["k_above_50"] >= 1 → True
    """

def check_conditions(row_data, conditions):
    """多條件 AND：全部滿足才回傳 True"""
    return all(check_condition(row_data, c) for c in conditions)
```

#### 核心流程

```python
def on_bar(self, symbol, bar_1m):
    """每收到一根 1m bar 觸發"""

    # 1. 每 5 分鐘自動 reload 策略（保留舊狀態）

    #    indicators.py 會算出 k_above_50, dif_above_dea, kd_cross 等所有欄位
    indicators = calculate_indicators(bars)

    observe_row = ...           # 觀察 TF 的最新 bar
```

#### 信號類型

| 信號 | 意義 | 狀態轉換 |
|------|------|----------|
| `ENTRY_SIGNAL` | 進場交叉信號觸發 | IDLE → WATCHING |
| `CLOSE_CANCEL` | 觀察期間 Close 條件觸發 | WATCHING → IDLE |

### 2.3 tf_aggregator.py — 即時 K 棒時間框架聚合器

**檔案：** `ib_dashboard/tf_aggregator.py`（218 行）

**功能：** 將 Databento 即時 1m bar 聚合為 3m/5m/15m/30m/1h/2h/3h/4h K 棒，所有 TF 從 9:30 ET 開盤時間對齊。

#### 聚合邏輯

```
收到 1m bar (例：10:03 TSLA)
  │
  ├─ 解析時間 → 轉換到 NY 時區
  ├─ mins_since_open = (bar_time - 9:30) in minutes
  ├─ 過濾盤前/盤後（只處理 9:30-16:00）
  │
  └─ 對每個 TF（3m, 5m, 15m, 30m, 1h, 2h, 3h, 4h）：
      │
      ├─ bar_index = mins_since_open // tf_minutes
      │   例：10:03 → mins=33 → 5m bar_index=6
      │
      ├─ 若 bar_index 改變（新 TF K 棒開始）：
      │   └─ 上一根 TF K 棒 → 存入 bars deque（maxlen=200）
      │
      └─ 更新當前 TF K 棒：
          ├─ open  = 第一根 1m 的 open
          ├─ high  = max(所有 1m 的 high)
          ├─ low   = min(所有 1m 的 low)
          ├─ close = 最新 1m 的 close
          └─ volume = sum(所有 1m 的 volume)
```

#### 記憶體結構

```python
_data = {
    'TSLA': {
        '5m': {
            'current': {ts, open, high, low, close, volume},  # 正在構建
            'current_index': 6,
            'current_date': date(2026, 3, 12),
            'bars': deque([...], maxlen=200)  # 已完成 K 棒
        },
        '15m': { ... },
        '1h': { ... },
    }
}
```

| 分類 | 欄位 | 計算 |

| **MACD** | `dif` | EMA(close,12) - EMA(close,26) |

| | `kd_cross` | 同上 |

| | `k_above_d`, `dif_above_dea` | 1 或 -1 |
| | `dif_above_zero`, `dea_above_zero` | 1 或 -1 |
| **交叉後計數** | `dif_up/down_since_cross` | 金叉/死叉後持續幾根 |
| | `k_up/down_since_cross` | 同上 |
| | `d_up/down_since_cross`, `j_up/down_since_cross`, `dea_up/down_since_cross` | 同上 |

| | `avg_low_sell` | low < prev HA-low → 1 |

## 七、安全機制

| 層級 | 機制 | 說明 |
|------|------|------|
| **IB 帳戶** | Paper Trading | `TRADING_MODE=paper`, port 4002 |
| **下單數量** | `default_shares=10` | 每筆約 $2,500（TSLA），可 API 調整 |
| **持倉上限** | `max_total_positions=5` | 最多 5 個策略同時持倉 |
| **重複防護** | `_inflight` set | 同一策略不會同時有兩筆訂單 |
| **全域開關** | `config['enabled']` | `POST /api/trading/config` 即時停用 |
| **IB 斷線** | Queue 保留 | 未連線時 order 放回 Queue 等重連 |
| **策略 reload** | 狀態保留 | reload 不中斷進行中的策略狀態 |
| **線程安全** | `threading.Lock` | 所有共享狀態用 Lock 保護 |
